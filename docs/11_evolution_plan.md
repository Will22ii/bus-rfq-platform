# Shuttle Partner RFQ · 발전 방향 및 추가 기능 계획

본 문서는 현재 MVP를 기준으로 **서비스적으로 더 발전시킬 방향**과 **추가 가능한 기능**을 계획만 정리한 것이다.  
**코드 수정은 하지 않으며**, 추후 개발 시 우선순위·범위 결정용으로 사용한다.

---

## 1. 현재 상태 요약

### 1.1 구현된 범위
- **흐름**: RFQ 생성 → 견적 제출 → 비교(마스킹) → 공급사 선택 → 완료 → 선택 공급사 연락처 공개
- **역할**: Requester / Supplier (겸용 계정 지원), 회사당 계정 1개
- **인증**: Supabase Auth (이메일+비밀번호), 관리자 공용 비밀번호(UI만)
- **제한**: 데스크톱 전용, 폐쇄 베타, 차량 계약·운행은 플랫폼 외부

### 1.2 문서상 미완/선택 사항
- Middleware 세션 확인 → 현재 클라이언트 리다이렉트로 대체
- Sidebar → 헤더로 대체
- 알림 → 목록만 구현, 발송/연동 최소
- 관리자 → 로그인 후 실제 기능(전체 조회·통계 등) 미연동

## 2. Phase 1: 서비스 안정화·보완

목표: 베타 운영 시 **즉시 문제가 될 수 있는 부분**을 보완.

| # | 항목 | 내용 | 비고 |
|---|------|------|------|
| 1.1 | **Middleware 세션** | 미인증 사용자를 라우트 진입 전에 /login으로 보내기. (main) layout 클라이언트 체크 대신 Next.js Middleware에서 세션 검사. | 보안·일관성 |
| 1.2 | **관리자 실제 인증** | /admin-login을 공용 비밀번호가 아닌, 관리자 전용 계정(Supabase Auth 또는 화이트리스트)으로 로그인. admin 전용 role 또는 플래그로 구분. | 운영 보안 |
| 1.3 | **알림 실사용** | RFQ 생성·견적 제출·취소·완료·공급사 선택 시 notifications 테이블 INSERT. 목록뿐 아니라 “읽음” 처리, 링크(해당 RFQ 상세/비교) 연결. | 사용성 |
| 1.4 | **동시 선택 정책 명시** | 공급사 선택 API의 “last-write wins” 정책을 이용자/운영 문서에 명시. 필요 시 “선택 시 확인 문구” 등으로 의도적 최종 선택 유도. | 운영 정책 |
| 1.5 | **에러 메시지 정리** | API·클라이언트 에러 메시지를 한국어로 통일하고, 자주 나오는 경우(마감 경과, 중복 제출 등) 사용자 안내 문구 정리. | 사용성 |

## 3. Phase 2: 사용성·운영 강화

목표: **일상 운영**과 **데이터 활용**을 쉽게 한다.

| # | 항목 | 내용 | 비고 |
|---|------|------|------|
| 2.1 | **RFQ 목록 검색·필터** | 제목·공연명·상태(open/in_review/completed/cancelled)·기간으로 필터. Requester는 “내가 만든 RFQ” 빠른 필터. | 사용성 |
| 2.2 | **관리자 기능 연동** | 관리자 로그인 후: 전체 RFQ 목록, 상태별 건수, 회사별 참여 현황, Success/Fail RFQ 통계(문서 13절). 필요 시 회사·공급사 정보 조회(마스킹 해제). | 운영 |
| 2.3 | **대시보드(선택)** | Requester: 내 RFQ 요약(접수중/심사중/완료). Supplier: 참여 가능 RFQ 수·제출 완료 건수. 관리자: 금주/금월 RFQ·완료율. | 사용성 |
| 2.4 | **내보내기** | RFQ 목록·상세·비교 결과를 CSV/엑셀 다운로드. (완료된 RFQ 기준으로 선택 공급사·가격 정리.) | 운영·보고 |
| 2.5 | **마감일·알림 강화** | 마감 D-1 등 특정 시점에 알림(이메일 또는 in-app). 설정은 관리자 또는 고정 규칙. | 참여율 |
| 2.6 | **Sidebar(선택)** | UI 스펙의 Sidebar 도입. RFQ 목록 / My RFQ / 알림 / 관리자 등 빠른 이동. | 사용성 |

## 4. Phase 3: 기능 확장

목표: **입찰 이후**까지 플랫폼 역할을 조금 넓히거나, **재사용·효율**을 높인다.

| # | 항목 | 내용 | 비고 |
|---|------|------|------|
| 3.1 | **RFQ 복사(템플릿)** | 완료/취소된 RFQ를 “복사”해 새 RFQ 초안 생성. 제목·날짜·노선 등 수정 후 제출. | 반복 요청 편의 |
| 3.2 | **노선/출발지 템플릿** | 자주 쓰는 노선 조합(출발지+도착지+시간대)을 저장해 RFQ 생성 시 선택. (테이블: route_templates 또는 회사별 저장.) | 사용성 |
| 3.3 | **출발지 관리** | 관리자가 출발지(departure_points) 추가·수정·비활성화. 현재는 시드만 있음. | 운영 |
| 3.4 | **간단 메시징(선택)** | RFQ 완료 후 Requester ↔ 선택 공급사 간 플랫폼 내 짧은 메시지(문의·확인). 계약은 여전히 외부. | 연계 |
| 3.5 | **평가/리뷰(선택)** | 완료된 RFQ에 대해 Requester가 공급사에 별점·한 줄 코멘트. 집계는 관리자 또는 Supplier 대시에만 노출. | 품질·선택 참고 |
| 3.6 | **요약 이메일** | RFQ 생성·마감·완료 시 Requester/Supplier에게 요약 메일(제목·상태·링크). Supabase Auth 이메일 또는 Resend 등. | 참여율 |

## 6. 제품 범위와의 관계

- **문서(01_product_context)** 에서 “차량 계약 및 운행 관리는 플랫폼 외부”로 되어 있으므로, **계약서 생성·서명·배차 실행**은 이 계획에서 제외**한다. (추후 제품 범위가 바뀌면 별도 기획.)
- **폐쇄 베타** 전제이므로, Phase 1·2를 먼저 하고 베타 피드백 후 Phase 3·4 순서를 정하는 것을 권장한다.

---

## 7. 우선순위 제안(요약)

| 순서 | 초점 | 대표 항목 |
|------|------|-----------|
| 1 | 안정화 | Middleware 세션, 관리자 인증, 알림 실사용 |
| 2 | 운영 | 목록 검색·필터, 관리자 통계·전체 조회, 내보내기 |
| 3 | 확장 | RFQ 복사, 노선 템플릿, 출발지 관리 |
| 4 | 정식 대비 | 감사 로그, 다중 계정(필요 시), 모니터링 |

위 순서로 계획만 수립했으며, 실제 개발 시에는 리소스와 베타 목표에 맞춰 항목을 골라 적용하면 된다.
