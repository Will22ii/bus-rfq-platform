# 디자인 개선 계획 (Design Improvement Plan)

본 문서는 **시각·UX 개선**을 위한 계획만 다룬다. **핵심 비즈니스 로직·API·데이터 구조는 변경하지 않는다.**

---

## 1. Primary 컬러 → 노랑

### 목표
- CTA(주요 행동 버튼)가 눈에 띄도록 **Primary를 노랑 계열**로 변경.
- 브랜드 포인트 컬러로 활용.

### 범위 (코드 변경 최소)
- **`src/app/globals.css`** (또는 shadcn 테마): `:root` 및 `.dark`의 `--primary`, `--primary-foreground` 값만 변경.
- 노랑 사용 시 가독성을 위해 **primary-foreground는 진한 색(검정/짙은 회색)** 유지.
- oklch 예시: primary 노랑 `oklch(0.85 0.15 95)` 등, 대비 맞춰 조정.

### 유의
- 상태 배지(파랑/주황/초록/회색)는 **그대로 유지** (Primary와 별개).
- 버튼 variant="default"만 primary 배경을 쓰므로, 다른 컴포넌트 로직 변경 없음.

## 2. 노선 표시 구간 정리 및 [수도권] / [지방] 그룹 표시

### 2.1 노선이 나오는 위치 (전부)

| 위치 | 파일 | 설명 |
|------|------|------|
| **RFQ 생성 1단계** | `src/components/rfq-create/route-selector.tsx` | 출발지(노선) 선택. 이미 **수도권/지방 탭**으로 구분됨. |
| **RFQ 생성 2단계** | `src/components/rfq-create/rfq-route-requirement-table.tsx` | 날짜별 노선 요건 테이블 — **노선** 컬럼. |
| **RFQ 상세** | `src/app/(main)/rfqs/[id]/page.tsx` | ① 노선 테이블(날짜 탭별) ② 가격 테이블(노선 기준) ③ Requester일 때 참여 공급사 견적 테이블 — 모두 **노선** 열 있음. |
| **RFQ 비교** | `src/app/(main)/rfqs/[id]/compare/page.tsx` | ① 노선별 공급 현황 ② 노선별 가격 — **노선** 열 있음. |

데이터: `departure_points`에 `region: 'metro' | 'local'` 있음. API에서 이미 `departure_points(id, name, region)` 조회 중.

### 2.2 적용 방향: 테이블에서 [수도권] / [지방] 묶어서 표시

- **RFQ 생성 2단계** (`rfq-route-requirement-table.tsx`):  
  노선 행을 `region` 기준으로 정렬하고, **섹션 구분** 삽입.  
  - 예: `[수도권]` 제목 행 또는 구분선 → 수도권 노선들 → `[지방]` 제목 행 또는 구분선 → 지방 노선들.  
  - 테이블 구조 유지. 행 데이터/제출 로직 변경 없음.

- **RFQ 상세** (`rfqs/[id]/page.tsx`):  
  노선 테이블, 가격(노선 기준) 테이블, 참여 공급사 견적 테이블에서 **노선 행을 region으로 정렬** 후 동일하게 `[수도권]` / `[지방]` 구간 표시.  
  - 날짜 탭·노선별 가격 키 등 기존 로직 유지.

- **RFQ 비교** (`compare/page.tsx`):  
  노선별 공급 현황·노선별 가격 테이블에서 **노선을 region으로 그룹**하여 `[수도권]` / `[지방]` 구간 표시.

- **RFQ 생성 1단계** (`route-selector.tsx`):  
  이미 수도권/지방 탭으로 나뉘어 있음. **추가 그룹 표시 불필요** (선택 시 탭으로 구분됨).

### 2.3 UI 스케치 (테이블)

```
| 노선     | 1회차 도착 | ... |
|----------|------------|-----|
| [수도권] |            |     |  ← 섹션 헤더 행 (colspan 또는 별도 스타일)
| 강남     | 10:00      | ... |
| 판교     | 10:30      | ... |
| [지방]   |            |     |
| 부산     | 09:00      | ... |
```

- 구현 시: 테이블 행으로 구분 행을 넣거나, `tbody`를 region별로 나누어 `<tr><td colspan="...">[수도권]</td></tr>` 형태로 처리. **정렬만 region 기준으로 하고, 데이터 스키마/API 변경 없음.**

## 3. 노선 선택 UX 개선 (현재: 체크박스 나열 → 스크롤 지옥)

### 3.1 현재 상태
- **파일:** `src/components/rfq-create/route-selector.tsx`
- 수도권/지방 **탭** + 탭 안에 **체크박스 나열** (flex-wrap).
- 출발지가 많으면 **스크롤·가독성 문제**.

### 3.2 개선 아이디어 (계획만, 구현 X)

| 방안 | 설명 |
|------|------|
| **A. 검색 + 필터** | 출발지 이름 검색 입력란 추가. "수도권" 탭 내에서도 검색으로 좁히기. |
| **B. 그룹 헤더 + 접기** | 탭 내에서 하위 그룹(예: 지역명 첫 글자, 또는 고정 그룹)으로 나누고, 섹션 접기/펼치기로 높이 절약. |
| **C. 멀티 Select(콤보)** | 체크박스 대신 "출발지 선택" 드롭다운(다중 선택 가능)으로 바꿔, 한 줄에 요약 표시. (선택된 N개 표시 후 클릭 시 목록.) |
| **D. 최근/자주 사용** | 자주 쓰는 출발지 상단 고정(설정 또는 휴리스틱). 나머지는 "더 보기"로 접기. |
| **E. 테이블형 선택** | 출발지를 테이블 한 열로 두고, 체크박스 열 추가. 테이블에 스크롤 + 헤더 고정으로 스캔 용이. |

**권장 검토 순서:** A(검색) → E(테이블) 또는 C(멀티 Select). 로직(선택된 id 배열, API 연동)은 유지하고 **표시 방식만** 바꾼다.

## 6. CTA(행동 버튼) 강화 — "행동이 약하다" 해소

### 6.1 현재 버튼 상태

| 버튼 | 위치 | 현재 스타일 | 문제 인식 |
|------|------|-------------|-----------|
| **RFQ 생성** | RFQ 목록 상단 | Button(default=primary) + Plus 아이콘 | Primary가 무채색이라 **눈에 안 띔**. |
| **다음: 2단계 노선 요건** | RFQ 생성 1단계 하단 | primary | **다음 단계**로 가는 전환인데, 단순 버튼 한 개라 흐름이 애매함. |
| **RFQ 생성하기** | RFQ 생성 2단계 | primary | 최종 제출인데 1단계와 동일 톤. |
| **견적 비교 보기** | RFQ 상세 | **variant="outline"** | 상세에서 비교로 가는 **핵심 진입점**인데 outline이라 **흐름 끊김**. |

### 6.2 개선 아이디어 (계획만)

| CTA | 개선 방향 |
|-----|-----------|
| **RFQ 생성** | Primary를 노랑으로 바꾸면 자동으로 강조. 추가로 버튼 크기 `lg`, 또는 "주요 액션" 영역(카드 상단 오른쪽)에 고정해 항상 보이게. |
| **다음 단계** | ① 1단계 하단 버튼을 **"다음: 2단계 노선 요건 입력 →"** 처럼 화살표·명확한 문구로. ② **스텝 인디케이터**(1단계 ● 2단계 ○)를 상단에 두어 "지금 1단계, 다음이 2단계"를 시각적으로 표시. ③ 버튼을 primary 유지해 노랑 적용 시 눈에 띄게. |
| **RFQ 생성하기** | 2단계에서만 쓰는 **최종 제출**이므로, primary + "생성하기" 문구 유지. 필요 시 아이콘(Check 등) 추가. |
| **견적 비교 보기** | **variant를 default(primary)** 로 변경하거나, "비교 보기"를 **카드/배너 형태**로 상단에 두어 "심사 중이면 여기서 비교·선택"을 유도. 상세 페이지 내에서 **고정 플로팅 버튼** 또는 상단 강조 영역으로 두는 방안 검토. |

### 6.3 원칙
- **주요 한 번만 할 행동**(RFQ 생성, 견적 제출, 비교 보기) → **Primary(노랑)** + 필요 시 크기/위치 강조.
- **보조/취소**(취소, 선택 해제 등) → secondary/outline 유지.
- **로직(라우팅, API 호출, 폼 제출)은 변경하지 않고**, variant·위치·문구·스텝 UI만 조정.

## 9. RFQ 리스트 카드형 전환

### 9.1 목표
- RFQ 목록을 **표(Table) 대신 카드형**으로 표시해 덜 딱딱하고 한 건씩 구분되게 보여 준다.
- **로직·API·라우팅은 그대로.** 클릭 시 기존처럼 상세(`/rfqs/[id]`)로 이동.

### 9.2 카드 레이아웃 (목표 스케치)

```
┌──────────────────────────────────────┐
│ 스트레이키즈 셔틀 차량 입찰           │  ← 제목 (title)
│ 인스파이어 아레나 · 2026.03.11        │  ← 행사장 · (견적 마감일 또는 제출일)
│                                      │
│                      [선택 완료]     │  ← 상태 배지 (우측 정렬)
└──────────────────────────────────────┘
```

- **1줄:** RFQ 제목 (`title`)
- **2줄:** 행사장 · 날짜 (날짜는 맥락에 따라 견적 마감일 또는 제출일)
- **우측 하단(또는 우측):** `StatusBadge` (견적 접수중 / 심사 진행중 / 선택 완료 / 요청 취소)
- 카드 전체 클릭 → `router.push(/rfqs/{id})` (기존과 동일)

### 9.3 적용 대상 (전부 찾은 위치)

| # | 페이지/위치 | 파일 | 현재 | 카드형 전환 |
|---|-------------|------|------|-------------|
| 1 | **RFQ 목록** | `src/app/(main)/rfqs/page.tsx` | Table: RFQ 제목, 행사장, 견적 마감일, 상태 | ✅ 카드 그리드로 전환. 데이터: `title`, `venue`, `quote_deadline_at`, `status`. 행사장 · 마감일 표시. |
| 2 | **My RFQ — 내가 생성한 RFQ** | `src/app/(main)/my/page.tsx` (탭 requester) | Table: 제목, 행사장, 견적 마감일, 상태 | ✅ 동일 카드형. 데이터: `title`, `venue`, `quote_deadline_at`, `status`. |
| 3 | **My RFQ — 내가 제출한 RFQ** | `src/app/(main)/my/page.tsx` (탭 supplier) | Table: RFQ 제목, 행사장, 제출일시, 상태 | ✅ 동일 카드형. 2줄: 행사장 · **제출일**(`submitted_at`). 상태 배지 + "제출 완료" 문구는 카드 안에 유지. |

위 세 곳이 **RFQ 리스트를 보여 주는 전부**이며, 모두 같은 카드 형태로 통일한다.

### 9.4 적용하지 않는 곳 (테이블 유지)

- **알림 목록** (`notifications/page.tsx`): 알림 타입·일시·읽음 — RFQ가 아니며 목록 구조가 달라 테이블 유지.
- **RFQ 상세 내부** (노선 테이블, 가격 테이블, 공급사 견적 테이블): 다열 데이터라 **테이블 유지**.
- **RFQ 비교** (노선별 공급·가격): 비교용 테이블이므로 **테이블 유지**.
- **RFQ 생성 2단계** (노선 요건 테이블): 입력 폼 테이블이므로 **테이블 유지**.

### 9.5 구현 시 참고
- 카드 리스트: `grid` 또는 `flex flex-wrap`으로 카드 배열 (예: `grid-cols-1 sm:grid-cols-2 lg:grid-cols-3`).
- 카드 컴포넌트: 기존 `Card` 또는 `div` + `border`/`rounded`/`shadow` 등.
- 데이터·API·`onClick` 경로는 **변경 없음**. 표시만 Table → 카드로 교체.

## 8. 제한 사항 (다시 강조)

- **API·DB·인증·알림 생성·RFQ 상태 전이** 등 핵심 로직은 **건드리지 않는다.**
- **디자인·스타일·레이아웃·문구·표시 순서(정렬/그룹)** 만 변경 대상.
- 노선 그룹 표시 시 **`departure_points.region`** 은 이미 있으므로 **데이터 스키마 변경 없음.**
