# Requester 본인 RFQ · 공급사 견적 표시 및 마스킹 검토

## 1. 요구사항 정리 (사용자 의도)

| 구분 | 요구사항 |
|------|----------|
| A | **내가 생성한 RFQ**에 들어갔을 때, **지금까지 공급에 참여한 공급사들이 제출한 견적**이 보여야 한다. |
| B | 그 견적은 **마스킹된 상태**로 표시된다 (공급사 이름·전화번호 마스킹). |
| C | **open** 상태에서도, 중간에 들어가면 **지금까지 참여한 공급사들의 공급가가 추가된 화면**이 나와야 한다. |
| D | **in_review(심사중)** 화면은 기한 내 참여한 **모든** 공급사 내용이 포함된 화면이다. |
| E | 공급사 이름·전화번호 마스킹은 **심사가 끝나기(completed)** 전까지 유지되고, **완료 후에는 선택된 공급사만** 연락처 공개. |

정리하면:

- **Requester가 본인 RFQ 상세에 들어갔을 때**
  - **open**: “지금까지 제출한 공급사” 견적 표시 (이름·전화 마스킹).
  - **in_review**: “기한 내 제출한 모든 공급사” 견적 표시 (마스킹).
  - **completed**: 선택된 공급사만 연락처 공개, 나머지는 마스킹 유지.
- **공통**: 견적(공급 대수·가격)은 보이되, **이름·전화는 completed + 선택 공급사만** 공개.

## 2. 현재 구현 검증

### 2.1 API

| API | Requester일 때 응답 | 마스킹 여부 |
|-----|---------------------|-------------|
| **GET /api/rfqs/[id]** (상세) | `rfq_supplier_submissions`에 `companies(id, name)` 포함. supply, prices, selections 모두 반환. | ❌ **마스킹 없음** – 회사명·전화가 그대로 내려감. (전화는 현재 상세 API에서 별도 조회하지 않음) |
| **GET /api/rfqs/[id]/compare** | `supplier_submissions`에 `company_name`, `public_phone` 포함. completed이고 선택된 공급사만 실제값, 나머지 `***`. | ✅ **마스킹 적용** (open/in_review 시 전부 `***`, completed 시 선택만 공개) |

**문제점**

- 상세 API는 Requester에게 **참여 공급사 목록·견적 데이터는 주지만**, 회사명·전화에 대한 **마스킹을 하지 않음**.
- 상세 API는 `public_phone`을 조회·반환하지 않음 (compare만 조회함).

### 2.2 UI (RFQ 상세 페이지)

| 상황 | 현재 동작 |
|------|------------|
| Requester가 본인 RFQ 상세 진입 | 노선 테이블(요청 내용) + «견적 비교 및 공급사 선택»(in_review/completed 시) + «RFQ 취소»(open/in_review 시)만 표시. |
| “참여한 공급사들의 제출 견적” 표시 | ❌ **없음** – `submissions`, `supply`, `prices`는 API로 받지만 **Requester용으로는 테이블/카드로 렌더링하지 않음**. |

**문제점**

- open 상태에서 Requester가 상세만 보면 “누가 얼마나 제출했는지”를 **전혀 볼 수 없음**.
- in_review에서도 상세 페이지에는 “참여 공급사 견적 요약”이 없고, **compare 페이지로만** 그 내용을 볼 수 있음.

### 2.3 compare 페이지

- Requester 전용, **in_review / completed**일 때 «견적 비교 및 공급사 선택»으로 진입.
- API가 이미 마스킹된 `company_name`, `public_phone`을 주므로, **compare 화면에서는 마스킹 정책 충족**.
- open에서 compare 진입 가능 여부는 스펙상 “in_review 이상에서만 버튼 노출”로 되어 있어, **open에서는 compare 진입 불가**가 현재 동작.

## 3. 요구사항과의 갭 요약

| # | 갭 | 심각도 |
|---|-----|--------|
| 1 | **상세 API (GET /api/rfqs/[id])** – Requester 응답에 회사명·전화 **마스킹 미적용** | 높음 (sealed bidding 위반 가능) |
| 2 | **상세 페이지** – Requester가 보는 “참여 공급사 견적” **블록 없음** (open/in_review/completed 공통) | 높음 (요구사항 A·C·D 미충족) |
| 3 | compare는 in_review/completed만 노출 → open 시 “지금까지 참여한 견적”은 **상세에서만** 보여줘야 함 | 중간 (상세에 블록 추가로 해결) |

## 4. 보완 방향 (안)

### 4.1 API: GET /api/rfqs/[id] (Requester 분기)

- **목표**: Requester일 때도 compare와 동일한 **마스킹 규칙** 적용.
- **방법**  
  - `rfq_supplier_submissions` 조회 후, 해당 `supplier_company_id`에 대해  
    - `companies(id, name)`  
    - `user_profiles(company_id, public_phone)` (동일 company 기준)  
    조회.
  - RFQ `status`와 `rfq_route_selections`로 “선택된 제출” 집계.
  - 응답의 `rfq_supplier_submissions`를 다음처럼 변환하여 반환:
    - `status !== 'completed'` → 모든 제출에 대해 `company_name`, `public_phone`을 `"***"` 등으로 마스킹.
    - `status === 'completed'` → `selected_supplier_submission_id`에 해당하는 제출만 실제 `company_name`, `public_phone` 반환, 나머지는 마스킹.
  - 기존 `rfq_routes`, `rfq_supplier_route_supply`, `rfq_supplier_route_prices`, `rfq_route_selections` 구조는 유지 (필요 시 필드명만 정리).

이렇게 하면 **상세 한 번만 써도** “참여 공급사 견적”을 마스킹된 형태로 안전하게 노출할 수 있음.

### 4.2 UI: RFQ 상세 페이지 (Requester)

- **목표**:  
  - **open**: 지금까지 제출한 공급사 견적 표시 (마스킹).  
  - **in_review / completed**: 기한 내 제출한 모든 공급사 견적 표시 (마스킹 → completed 시 선택만 연락처 공개).
- **방법**  
  - `isRequester && (status === 'open' || status === 'in_review' || status === 'completed')` 일 때  
    **“참여 공급사 견적”** (또는 “제출된 견적”) 카드/섹션 추가.
  - 내용:  
    - (선택) 공급사 수 요약 문구.  
    - 노선별·공급사별 공급 대수·가격 테이블 (이미 상세 API에서 오는 supply/prices 활용).  
    - 공급사 식별은 **마스킹된 이름** (예: 공급사 A, B 또는 `***`)만 표시.  
  - in_review / completed일 때는 기존처럼 «견적 비교 및 공급사 선택» 버튼 유지 → compare에서 선택·완료 처리.

이렇게 하면 **상세만으로** “지금까지 참여한 견적이 추가된 화면” 요구사항을 충족하고, compare는 “선택/완료”용으로 유지 가능.

### 4.3 compare 페이지

- **역할**: in_review에서 “노선별 공급사 선택” + completed에서 “선택된 공급사 연락처 확인”.
- **변경**:  
  - API가 이미 마스킹 적용하므로 **로직 변경 최소화**.  
  - 필요 시 “open에서도 compare 진입 허용할지”만 정책 결정 (현재 제안: **open에서는 상세에만 “참여한 견적” 표시**, compare는 in_review부터).

## 5. 확인이 필요한 사항 (정렬용)

아래 세 가지는 UI/UX와 구현 범위에 영향을 주므로, 방향을 정한 뒤 수정하는 것이 좋습니다.

---

### 5.1 상세 페이지 “참여 공급사 견적” 블록 – 어떤 형태로 보여줄지

**선택지 A: open / in_review / completed 모두 같은 형태의 테이블**

- **의미**: RFQ 상태와 관계없이, 상세 페이지에 항상 **동일한 “참여 공급사 견적” 테이블**을 둔다.
  - 노선 × 공급사별로 공급 대수·왕복가·편도가 보이는 표.
  - open일 때는 “지금까지 제출한 공급사”만, in_review/completed일 때는 “기한 내 제출한 모든 공급사”가 행으로 들어감.
- **장점**
  - Requester가 상세만 보고도 **항상 같은 위치**에서 견적 현황을 파악할 수 있음.
  - 구현이 단순함 (한 종류의 테이블 컴포넌트만 유지).
- **단점**
  - open 초반에는 제출이 0건일 수 있어, 빈 테이블이 자주 노출됨. “아직 제출된 견적이 없습니다” 문구로 보완 가능.

**선택지 B: open일 때는 “제출 건수 + 요약”만, in_review부터 전체 테이블**

- **의미**:
  - **open**: “참여 공급사 N명”, “제출된 견적 N건” 같은 **숫자 요약**만 표시. (선택) “상세는 마감 후 비교 화면에서” 안내.
  - **in_review / completed**: 상세 페이지에 **전체 테이블**(노선·공급사·공급 대수·가격) 표시.
- **장점**
  - open 구간에서는 “누가 얼마”를 숨기는 느낌을 더 줄 수 있음 (실제 데이터는 API에서 마스킹해도, 화면에는 건수만).
  - “마감 후 비교에서 보세요”로 compare 페이지 역할이 분명해짐.
- **단점**
  - “지금까지 참여한 공급사들의 공급가가 추가된 화면”을 **open에서도** 보여주고 싶다면 요구와 어긋남.
  - 상태별로 UI가 달라져 구현·테스트가 늘어남.

**정리**:  
- **“open에서도 지금까지 제출된 견적을 테이블로 보고 싶다”** → **선택지 A**.  
- **“open에서는 건수/요약만, 상세 테이블은 마감 후”** → **선택지 B**.

---

### 5.2 compare 진입 시점 – open에서도 들어갈 수 있게 할지

**현재**: «견적 비교 및 공급사 선택» 버튼은 **in_review / completed**일 때만 노출 → compare 페이지는 “마감 후” 전용.

**선택지 A: open에서도 compare(읽기 전용) 진입 허용**

- **의미**:
  - open 상태에서도 “견적 비교 보기” 같은 버튼을 두고, compare 페이지로 이동 가능.
  - compare 페이지 안에서는 **선택/완료 버튼은 비활성 또는 미노출**, “지금까지 제출된 견적만 보기” 용도.
- **장점**
  - “비교” 화면이 **한 곳**이어서, 사용자가 “견적 비교는 여기”로만 기억하면 됨.
  - open일 때도 노선별·공급사별 비교 테이블을 그대로 활용 가능.
- **단점**
  - compare 페이지에 **open일 때 / in_review일 때** 분기 필요 (선택·완료 버튼 노출 여부, 문구 등).
  - “비교”와 “선택/완료”가 한 페이지에 섞여 있어, 화면 설명이 조금 더 필요할 수 있음.

**선택지 B: open에서는 상세의 “참여 공급사 견적” 블록만 제공**

- **의미**:
  - **open**: 상세 페이지의 “참여 공급사 견적” 블록에서만, 지금까지 제출된 견적을 테이블로 표시. compare 버튼은 안 보임.
  - **in_review / completed**: 기존처럼 «견적 비교 및 공급사 선택» 노출 → compare에서 선택·완료.
- **장점**
  - 역할이 명확함. “상세 = 현황 보기”, “compare = 마감 후 비교·선택·완료”.
  - compare 페이지는 “선택 가능한 상태”에서만 들어가므로, 버튼/문구 분기가 단순함.
- **단점**
  - “참여 공급사 견적”이 **상세**와 **compare** 두 곳에 비슷한 테이블로 나올 수 있어, 디자인/구현을 맞춰야 함.

**정리**:  
- **“open에서도 비교 페이지 레이아웃으로 보고 싶다”** → **선택지 A**.  
- **“open은 상세에서만 견적 보기, 비교/선택은 마감 후에만”** → **선택지 B** (문서 4절 기본 제안과 동일).

---

### 5.3 마스킹 표기 – `***` vs “공급사 A/B” 익명 라벨

**선택지 A: `***` 통일**

- **의미**: 회사명·전화번호 자리에 모두 **`***`** (또는 동일한 마스킹 문자)만 표시.
- **장점**
  - 구현이 단순하고, “숨김”이 시각적으로 분명함.
  - 별도 라벨 규칙이 없어, 공급사 수가 많아도 동일하게 처리 가능.
- **단점**
  - 테이블에서 “공급사 1, 2, 3”을 **구분**하기 어렵다. 열 제목이 “공급사 A / B / C”가 아니라 “*** / *** / ***”처럼 되면, 어떤 열이 어떤 제출인지 구분이 안 됨.
  - 따라서 **열(또는 행) 구분**은 “공급사 1, 2, 3” 같은 **위치 기반 라벨**이 여전히 필요할 수 있음.

**선택지 B: “공급사 A / B / C” 같은 익명 라벨**

- **의미**:
  - 제출 순서나 목록 순서대로 “공급사 A”, “공급사 B”, “공급사 C”처럼 **익명 라벨** 부여.
  - 회사명·전화는 마스킹(`***`) 유지. completed 후 **선택된 공급사만** “OO 회사”, “010-****-****” 등으로 전환.
- **장점**
  - 테이블에서 “A가 이 가격, B가 이 가격”처럼 **비교하기 쉽고**, “공급사 B로 선택” 같은 의사결정도 자연스러움.
  - 완료 후 “공급사 B = 서울고려”처럼 한 번만 매핑해 주면, 사용자 이해가 쉬움.
- **단점**
  - 라벨 규칙(순서 고정, A/B/C 개수 등)을 정해야 하고, UI/API에서 순서를 맞춰야 함.

**정리**:  
- **테이블에서 공급사를 구분할 필요가 거의 없고, “숨김”만 강조하면 됨** → **선택지 A** (이때 열/행 구분은 “제출 1, 2, 3” 등 위치로 처리 가능).  
- **노선별로 “어느 공급사가 얼마”를 비교·선택하는 UX가 중요** → **선택지 B**가 유리.

## 8. 구현 완료 사항

1. **상세 API (GET /api/rfqs/[id])** – Requester 분기에서 제출 목록을 `submitted_at` 오름차순으로 조회, 회사명·전화 마스킹(***) 및 completed 시 선택된 공급사만 실제값 반환. `public_phone` 조회·반환 및 `supplier_label`(공급사 A/B/C) 부여.
2. **상세 페이지** – “참여 공급사 견적” 카드 추가 (노선×공급사 공급/가격 테이블, 공급사 A/B/C 라벨, 0건 시 “아직 제출된 견적이 없습니다.”). open일 때 «견적 비교 보기», in_review/completed일 때 «견적 비교 및 공급사 선택» 버튼 노출.
3. **compare API** – 제출 목록 `submitted_at` 오름차순, 응답에 `supplier_label`(공급사 A/B/C) 포함.
4. **compare 페이지** – status가 open일 때 읽기 전용(공급사 선택 컬럼·RFQ 완료하기 미노출), 제목 “견적 비교 보기”로 표시. 공급사 컬럼에 `supplier_label` 사용.
